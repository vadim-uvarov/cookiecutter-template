# Coding Standards

## Clean code
- Avoid very large functions. It's always better to break down a large function into several smaller ones with clear naming.

## Commits
- Use commitizen format for commit messages
- Mention yourself as a co-author
- Commit along the way after completing each logical unit of work
- Combine related code changes into a single commit
- Avoid commiting to main, instead create a new branch and then open a PR

## Function Naming
- Start all function and method names with action verbs.
- Prefer full variable names for readability (e.g., `response` instead of `resp`)
- Python: separate code from the inline comment with two spaces

## Auto-Formatting
- Python: use `ruff` and `mypy` for auto-formatting, linting and type checking
- Terraform: use `terraform fmt` for auto-formatting

## Comments and docstrings
- Use Google-style
- For inline and line comments, focus on explaining why the referred code exists, not how it works
- Add inline comment if a purpose of a block or a line of code is not easy to understand
- Add a line comment if a purpose of a line of code is not easy to understand
- Docstring of a public function or method: add description of what it does, then brief description of each parameter and return value
- Docstring of a private functions or methods: if function name is not self-evident, then add a brief description of what it does. Optionally add a brief description of each parameter and return value, if it's not easy to infer from their names
- Classes: add a docstring with brief description

## Tests

- Python: use pytest framework for testing

### Unit tests
- Write at least one unit test for each function that is not very simple
- Don't test obvious and simple functions
- Test only the project logic, don't test functions imported from external packages
- Use best practices of unit testing:
    - have at least one smoke test
    - ideally, have a test for each of the function if-branches
    - test border cases
    - if function is numeric, then test border cases and the "normal" case


### Integration tests
- use the best practices of integration testing
- have at least one smoke test


# Pull Request

## Addressing pull request comments
- When addressing PR comments, group similar changes and commit them into separate commits
- After finishing addressing PR comment, reply to the comment with a message that it's done 
    - provide summary on what changed
    - don't mention commit hash
    - mention that this reply was generated by Claude Code

## Opening a pull request
- Open a new pull request after a big change to the code
- Avoid mixing several unrelated tasks inside one pull request
- If a previous pull request is not yet merged into main, and you need to build a new task on top of these changes, then create a new branch from the previous branch, make changes and open a new pull request which targets the previous pull request
